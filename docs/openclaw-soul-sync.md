# OpenClaw SOUL 同步与安全对账

## 概述

Multi-Agent-Flow 现已补齐 OpenClaw Agent 的 `SOUL.md` 读取、导入与连接后同步流程，重点解决以下两个问题：

- 连接 OpenClaw 并同步 agent 信息后，部分 agent 的 `soul.md` 没有正确读取到项目中
- 本地项目里的 SOUL 与 OpenClaw 中的 SOUL 不一致时，缺少一套保守且可解释的处理策略

这次能力升级后，系统会优先从真实的 OpenClaw SOUL 文件路径读取内容，并在连接阶段基于会话备份执行一次安全对账，避免误覆盖远端内容。

## 解决了什么

- 统一了 OpenClaw SOUL 文件的查找逻辑，不再只依赖单一路径
- 识别结果中记录真实 `soulPath`，导入和后续同步都尽量使用同一来源
- 在连接 OpenClaw 时，先用连接前备份作为远端真值做 SOUL 对账，再决定是否更新项目内容
- 为每个 agent 记录最近一次导入的 SOUL 基线，后续可以判断“谁改过”
- 冲突场景不再自动替用户做决定，保留人工判断空间

## SOUL 路径识别策略

系统现在会统一搜索以下位置：

- `root/SOUL.md`
- `root/soul.md`
- `root/agent/SOUL.md`
- `root/agent/soul.md`
- `root/private/SOUL.md`
- `root/private/soul.md`

同时支持有限层级的祖先目录回退，因此即使 agent 工作目录、私有目录、workspace 目录结构不同，也能更稳定地命中真实 SOUL 文件。

识别到的 OpenClaw agent 会额外保存：

- `directoryPath`
- `workspacePath`
- `soulPath`

其中 `soulPath` 会作为后续导入、镜像写回和会话对账时的优先参考路径。

## 连接阶段的安全对账

连接 OpenClaw 时，系统会先创建一份当前 OpenClaw 会话备份，并以这份备份中的 SOUL 作为远端基准，然后把项目内容写入 session mirror，再同步到当前 OpenClaw 会话。

这一步的核心目标是：

- 尽可能修复“项目里没读到真实 SOUL 内容”的问题
- 避免在不明确的情况下直接覆盖 OpenClaw 里的远端 SOUL
- 为后续冲突处理保留清晰的基线信息

## 对账规则

每个 agent 都会记录最近一次导入基线：

- `lastImportedSoulHash`
- `lastImportedSoulPath`
- `lastImportedAt`

连接时按以下规则处理：

### 1. 本地与 OpenClaw 一致

- 判定为“确认一致”
- 不改动 SOUL 内容
- 刷新同步基线

### 2. 本地仍是默认模板或占位内容

- 判定为“自动更新”
- 用 OpenClaw 中的真实 SOUL 替换本地内容
- 刷新同步基线

### 3. 只有 OpenClaw 发生变化

- 判定为“自动更新”
- 本地仍等于上次基线，说明用户未改本地
- 自动把 OpenClaw 最新 SOUL 更新到项目中

### 4. 只有本地发生变化

- 判定为“保留本地”
- OpenClaw 仍等于上次基线，说明远端未改
- 保留本地 SOUL，不自动回滚

### 5. 本地与 OpenClaw 都发生变化

- 判定为“冲突，待人工处理”
- 不自动合并，不自动覆盖项目最终内容
- 当前连接会优先保留远端备份内容进入会话镜像，避免系统在后台替用户把远端覆盖掉

### 6. 首次同步即发现两边不一致

- 判定为“首次同步冲突”
- 说明历史上没有可用基线
- 记录冲突并等待用户判断

### 7. 无法在备份中定位 SOUL 来源

- 判定为“缺少源文件”
- 不进行危险覆盖
- 记录 warning，提示用户补查路径或 agent 结构

## 保守策略说明

本次实现的核心原则是“保守覆盖”：

- 能确认远端变更且本地未改时，才自动覆盖本地
- 能确认只有本地改动时，保留本地
- 只要进入冲突或来源不明确场景，就不自动替用户做最终选择

尤其在“双方都改过”或“首次同步已不一致”这两类高风险场景中，系统不会把“看起来更像正确”的一方直接写死，而是把冲突显式暴露出来。

## 用户可见结果

连接完成后，系统会输出一份 SOUL 同步摘要，例如：

- 自动更新了多少个 agent
- 确认一致了多少个 agent
- 保留本地了多少个 agent
- 还有多少个冲突待人工处理
- 多少个 agent 未找到源文件

对于冲突与缺源，还会额外记录 warning 级别日志，方便后续排查。

## 带来的收益

- 更多 OpenClaw agent 的真实 SOUL 内容可以被正确读取到项目内
- 导入、编辑、连接、镜像写回之间使用同一套 SOUL 路径认知，减少错位
- 同步策略从“可能静默覆盖”升级为“带基线的可解释对账”
- 冲突场景不会再在后台悄悄覆盖远端内容
- 后续如果要补做人工冲突处理界面，已经具备必要的基线与状态数据
